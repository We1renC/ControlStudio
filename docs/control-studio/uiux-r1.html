1｜設計工作流程（方法論）

本章記錄整套 UI/UX 設計的推導邏輯，供後續迭代、交接、或新功能評估時遵循。

1.1 設計起點：從功能廣度到使用者需求

ControlStudio 在 P29–P34 開發完成後，已具備 333 個 exported symbols，涵蓋：

領域符號數代表能力
control/188PID、LQR、H∞、MPC、自適應、非線性、識別
math/53矩陣代數、ODE、優化、SVD
analysis/24時域/頻域/根軌跡/相平面分析
ui/41主題、元件、圖表、可及性
utils/14格式化、LaTeX、codegen

功能已相對完整，但缺乏統一的使用者視角。設計改善從「使用者會帶著什麼目標來用這個工具」出發，而非從功能清單反推介面。

1.2 使用者族群識別（5 大方向）

識別方法：角色動機分析

對每一類潛在使用者，問三個問題：

  1. 他們的主要任務是什麼？（Primary Job-to-be-done）
  2. 他們最常在哪個步驟卡住？（Pain point）
  3. 他們最在意哪個輸出？（Desired outcome）
族群識別矩陣：

族群              主要任務                  卡住點                   期望輸出
────────────────────────────────────────────────────────────────────────────
A 控制設計者      設計滿足規格的控制器        調參反覆、看不到規格差距    可驗證的控制器
B 分析研究者      深入理解系統行為             圖表不夠精細、難以比較      可發表的數據/圖
C 學習者          建立控制理論直覺            不懂參數意義、缺乏引導      「啊，我懂了！」
D 實作工程師      產出可部署程式碼            格式轉換、平台適配          可執行的程式碼
E 技術決策者      審查設計品質                看不懂圖、缺乏摘要          決策用報告

族群邊界原則：

  • 同一人可能跨族群（研究員也會生成程式碼），但設計以主要動機分類
  • 功能不必每個族群都看到；導覽與預設視圖按族群分流

1.3 需求三層展開法

每個族群的需求按三層展開，確保從「為什麼用」到「用什麼」有清晰的推導鏈：

層 1 — 大方向（族群）
  └─ 層 2 — 中需求（子目標，3–5 個）
       └─ 層 3 — 功能（具體 UI 功能，每個中需求 3–5 個）

展開規則：

  • 中需求必須是可驗收的使用者目標（不是技術描述）
  • 功能必須對應至少一個現有模組或明確的新元件
  • 每個功能寫使用者故事：「身為 [族群]，我想 [動作]，以便 [效益]」

展開驗證：問「如果這個功能不存在，哪個族群會受阻？」如果答案是「沒有人」，則刪除此功能。

1.4 功能優先排序法（影響力 × 工程成本矩陣）

          工程成本
          低(1-2d)    中(3-4d)    高(5+d)
影響   高  ┌─────────┬─────────┬─────────┐

影響力評估（三問）：
  Q1. 影響幾個族群？（1族群=低，3+族群=高）
  Q2. 解決的是核心任務還是邊緣需求？
  Q3. 是否解鎖其他功能（基礎設施型）？

優先規則：

  • P1：全局架構類（F 群）+ 各族群最高頻操作
  • P2：單族群深度功能，但有明確現有模組可用
  • P3：專化功能，依資源彈性排入

A1-1｜系統輸入精靈

使用者故事

身為控制設計者，我想用引導式表單建立系統模型，不必記憶格式，系統自動驗證維度與穩定性。

觸發位置

  • SideNav「建模 → 新增系統」
  • TopBar 系統名稱下拉「+ 新增」
  • 空白工作區的「立即建模」CTA 按鈕

互動流程

Step 1 — 選擇模型類型
  [TF 傳遞函數] [SS 狀態空間] [ZPK 零極點增益]
  ↓ 點選後下方顯示對應輸入區

Step 2 — 輸入係數
  TF：分子多項式 / 分母多項式（兩行 input，示例 placeholder）
  SS：A / B / C / D 矩陣（textarea，支援空格分隔或 [1 2; 3 4] 格式）
  ZPK：零點列表 / 極點列表 / 增益 k

  ➜ 右側即時 LaTeX preview（呼叫 tfToLatex / renderTfLatex）

Step 3 — 模型健康診斷（即時）
  ✓ 維度一致（A 是 n×n，B 是 n×m，…）
  ✓ 穩定性（呼叫 analyzeStability → routhTable）
  ✓ 可控性（controllabilityMatrix → matRank）
  ✓ 可觀性（observabilityMatrix → matRank）
  → 四項 badge 即時更新

Step 4 — 命名與確認
  [系統名稱 input] [加入工作區] [取消]

Modal：寬 540px，高自適應（max 85vh），backdrop blur 8px

  • formField() — 每個係數輸入框（含 label + hint）
  • badge() — 四項健康診斷狀態
  • alert() — 維度錯誤時顯示 error alert
  • button() — 確認、取消

資料介面

輸入：使用者填入的係數字串

輸出：

  type: 'tf' | 'ss' | 'zpk',

使用者故事

觸發位置

互動流程

1. 資料輸入（二擇一）
   [貼上 CSV 文字] 或 [拖放 .csv 檔案]
   格式：time, output（單輸入步階）或 time, input, output

2. 圖表預覽
   → 即時繪製量測數據（buildSVGChart，散點 + 折線）

3. 選擇識別方法
   [ARX] [ARMAX] [OE] [子空間 N4SID] → 顯示各方法說明 tooltip

4. 設定參數
   na（A 多項式階數）、nb（B 多項式階數）、nk（延遲步數）
   → 「自動選階」按鈕（呼叫 autoARXOrder / autoModelOrder）

5. 執行識別 → StatusBar 顯示進度
   → 結果：殘差圖 + 白化檢驗 badge + 擬合度 %
   → [接受此模型] → 進入系統輸入精靈 Step 3（已填入 ss/tf）

視覺規格

Page：右窗格參數設定，左窗格圖表
資料輸入區：高 120px textarea，虛線 border，拖放高亮（--accent border）
方法選擇：radio 卡片，4 列，每卡片含名稱 + 一行說明
殘差圖：buildSVGChart，高 160px，自相關圖
擬合度：大字 badge，90%+ 綠，75-89% 黃，<75% 紅

依賴模組

identifyARX, identifyARMAX, identifyOE, identifySubspace, autoARXOrder, autoModelOrder, residualWhitenessTest, crossCorrelationTest, buildSVGChart, toCSV

驗收標準

  • [ ] CSV 拖放可解析時間+輸出兩欄格式
  • [ ] 自動選階在 500ms 內完成（n ≤ 10）
  • [ ] 白化檢驗 p-value 以顏色呈現（≥0.05 綠）
  • [ ] 「接受此模型」帶著參數跳轉到精靈 Step 3

  根據起始時間或第一個非零輸出對齊兩條曲線

  RMSE = sqrt(mean((y_sim - y_meas)²))
  最大誤差 max|y_sim - y_meas|
  相關係數 r（越接近 1 越好）

判斷：
  RMSE / 穩態值 < 5% → ✓ 模型與硬體吻合
  > 5% → ⚠ 建議重新 SysID
