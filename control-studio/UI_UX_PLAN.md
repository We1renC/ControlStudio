> 本章記錄整套 UI/UX 設計的推導邏輯，供後續迭代、交接、或新功能評估時遵循。

### 1.1 設計起點：從功能廣度到使用者需求

ControlStudio 在 P29–P34 開發完成後，已具備 333 個 exported symbols，涵蓋：

| 領域 | 符號數 | 代表能力 |
|------|-------|---------|
| control/ | 188 | PID、LQR、H∞、MPC、自適應、非線性、識別 |
| math/ | 53 | 矩陣代數、ODE、優化、SVD |
| analysis/ | 24 | 時域/頻域/根軌跡/相平面分析 |
| ui/ | 41 | 主題、元件、圖表、可及性 |
| utils/ | 14 | 格式化、LaTeX、codegen |

功能已相對完整，但缺乏統一的使用者視角。設計改善從「**使用者會帶著什麼目標來用這個工具**」出發，而非從功能清單反推介面。

### 1.2 使用者族群識別（5 大方向）

**識別方法：角色動機分析**

對每一類潛在使用者，問三個問題：
1. **他們的主要任務是什麼？**（Primary Job-to-be-done）
2. **他們最常在哪個步驟卡住？**（Pain point）
3. **他們最在意哪個輸出？**（Desired outcome）

```
族群識別矩陣：

族群              主要任務                  卡住點                   期望輸出
────────────────────────────────────────────────────────────────────────────
A 控制設計者      設計滿足規格的控制器        調參反覆、看不到規格差距    可驗證的控制器
B 分析研究者      深入理解系統行為             圖表不夠精細、難以比較      可發表的數據/圖
C 學習者          建立控制理論直覺            不懂參數意義、缺乏引導      「啊，我懂了！」
D 實作工程師      產出可部署程式碼            格式轉換、平台適配          可執行的程式碼
E 技術決策者      審查設計品質                看不懂圖、缺乏摘要          決策用報告
```

**族群邊界原則**：
- 同一人可能跨族群（研究員也會生成程式碼），但設計以**主要動機**分類
- 功能不必每個族群都看到；導覽與預設視圖按族群分流

---

### 1.3 需求三層展開法

每個族群的需求按三層展開，確保從「為什麼用」到「用什麼」有清晰的推導鏈：

```
層 1 — 大方向（族群）
  └─ 層 2 — 中需求（子目標，3–5 個）
       └─ 層 3 — 功能（具體 UI 功能，每個中需求 3–5 個）
```

**展開規則**：
- 中需求必須是**可驗收的使用者目標**（不是技術描述）
- 功能必須對應**至少一個現有模組**或明確的新元件
- 每個功能寫**使用者故事**：「身為 [族群]，我想 [動作]，以便 [效益]」

**展開驗證**：問「如果這個功能不存在，哪個族群會受阻？」如果答案是「沒有人」，則刪除此功能。

---

### 1.4 功能優先排序法（影響力 × 工程成本矩陣）

```
          工程成本
          低(1-2d)    中(3-4d)    高(5+d)
影響   高  ┌─────────┬─────────┬─────────┐
力         │  P1 ★★★ │  P1 ★★  │  P2 ★   │
       中  │  P1 ★★  │  P2 ★★  │  P3 ★   │
       低  │  P2 ★   │  P3 ★   │  不做    │
           └─────────┴─────────┴─────────┘

影響力評估（三問）：
  Q1. 影響幾個族群？（1族群=低，3+族群=高）
  Q2. 解決的是核心任務還是邊緣需求？
  Q3. 是否解鎖其他功能（基礎設施型）？
```

**優先規則**：
- P1：全局架構類（F 群）+ 各族群最高頻操作
- P2：單族群深度功能，但有明確現有模組可用
- P3：專化功能，依資源彈性排入

### 1.5 設計 → 實作映射流程

```
使用者故事
    │
    ▼
確認現有模組能否覆蓋 70%+ 邏輯？
    │
    ├─ YES → 設計薄層 UI（呼叫現有函數）
    │
    └─ NO  → 列入新元件需求（§11）
                  │
                  ▼
            評估元件通用性：
              能否被 2+ 功能共用？
                YES → 加入 js/ui/ 擴充清單
                NO  → inline 實作，不抽元件
```

**設計 Token 優先**：所有視覺規格必須引用 `theme.js` 的 CSS 變數（`--bg`, `--surface`, `--accent`…），不使用 hardcode 色碼，確保三主題一致性。

### 1.6 功能規格卡標準格式

每個功能依下列模板撰寫，確保資訊完整可執行：

```markdown
### X#-#｜[功能名稱]

**使用者故事**：身為 [族群]，我想 [動作]，以便 [效益]。

**觸發位置**：[導覽路徑 / 按鈕位置]

**互動流程**
1. [步驟]
2. [步驟]

**視覺規格**
[尺寸、色號、元件描述]

**元件組成**：[components.js / a11y.js 中的函數]

**依賴模組**：[js/control 或 js/analysis 中的函數]

**驗收標準**
- [ ] [可測量的完成條件]
```

缺少任何一欄 → 該功能規格不完整，不可進入開發。

### 1.7 迭代驗收循環

```
規格完成
    │
    ▼
實作（引用現有模組）
    │
    ▼
對照驗收標準逐條勾選
    │
    ├─ 全部通過 → 標記功能狀態為 Done，更新 ROADMAP.md
    │
    └─ 有項未過 → 回到規格，判斷是規格問題還是實作問題
                  規格問題 → 修改規格 + 重新評審
                  實作問題 → 修 bug + 重測
```

**回歸測試**：每個 P1 功能完成後，跑 `bash scripts/run_all_verify.sh` 確認既有 49 個驗證腳本仍全部通過。

### A1-1｜系統輸入精靈

**使用者故事**
身為控制設計者，我想用引導式表單建立系統模型，不必記憶格式，系統自動驗證維度與穩定性。

**觸發位置**
- SideNav「建模 → 新增系統」
- TopBar 系統名稱下拉「+ 新增」
- 空白工作區的「立即建模」CTA 按鈕

**互動流程**
```
Step 1 — 選擇模型類型
  [TF 傳遞函數] [SS 狀態空間] [ZPK 零極點增益]
  ↓ 點選後下方顯示對應輸入區

Step 2 — 輸入係數
  TF：分子多項式 / 分母多項式（兩行 input，示例 placeholder）
  SS：A / B / C / D 矩陣（textarea，支援空格分隔或 [1 2; 3 4] 格式）
  ZPK：零點列表 / 極點列表 / 增益 k

  ➜ 右側即時 LaTeX preview（呼叫 tfToLatex / renderTfLatex）

Step 3 — 模型健康診斷（即時）
  ✓ 維度一致（A 是 n×n，B 是 n×m，…）
  ✓ 穩定性（呼叫 analyzeStability → routhTable）
  ✓ 可控性（controllabilityMatrix → matRank）
  ✓ 可觀性（observabilityMatrix → matRank）
  → 四項 badge 即時更新

Step 4 — 命名與確認
  [系統名稱 input] [加入工作區] [取消]
```

**視覺規格**
```
Modal：寬 540px，高自適應（max 85vh），backdrop blur 8px
Header：「新增系統模型」H2，右上 ✕
步驟 indicator：3 個圓點（Step 1/2/3），active 填滿 --accent
Step 卡片：--surface 底色，16px padding，6px border-radius
係數 input：font-family: monospace，14px，--code-bg 底色
LaTeX preview：高 56px，--surface-2，置中對齊，color:--accent-blue
診斷 badge 列：4 個 badge 水平排列，wrap，8px gap
Footer：[取消 ghost button] [加入工作區 primary button]
```

**元件組成**
- `tabs()` — TF / SS / ZPK 切換
- `formField()` — 每個係數輸入框（含 label + hint）
- `badge()` — 四項健康診斷狀態
- `alert()` — 維度錯誤時顯示 error alert
- `button()` — 確認、取消
- `renderLatex()` / `tfToLatex()` — 即時數學預覽

**資料介面**

輸入：使用者填入的係數字串
輸出：
```javascript
{
  type: 'tf' | 'ss' | 'zpk',
  name: string,
  data: TransferFunction | MIMOStateSpace | ZPK,
  health: { stable, controllable, observable, minPhase }
}
```

**狀態機**
```
idle → typing（input 變化）→ validating（防抖 150ms）→ valid | invalid
valid → confirmed（點擊「加入」）→ 關閉 modal，更新 TopBar 系統名稱
invalid → 顯示 error alert，阻止確認
```

**鍵盤支援**
- `Tab` 在欄位間移動
- `Enter` 在最後一個欄位觸發確認
- `Escape` 關閉 modal
- `Ctrl+1/2/3` 切換 TF/SS/ZPK tab

**依賴模組**
`TransferFunction`, `stateSpaceToTransferFunction`, `analyzeStability`, `routhTable`, `controllabilityMatrix`, `observabilityMatrix`, `matRank`, `tfToLatex`, `renderTfLatex`, `polyroots`, `parseMatrixInput`

**驗收標準**
- [ ] 輸入非方陣 A → error alert 阻擋確認
- [ ] 輸入不穩定系統 → warning badge（不阻擋）
- [ ] LaTeX preview 在輸入後 150ms 內更新
- [ ] 三種模式欄位正確切換，前一步驟資料清除
- [ ] 加入後 TopBar 顯示系統名稱，規格 badge 更新

### A1-2｜SysID 一鍵入口

**使用者故事**
我有實驗量測的步階響應 CSV 資料，想直接貼入或上傳就能得到估計的系統模型。

**觸發位置**
SideNav「建模 → SysID 入口」/ 系統輸入精靈 Step 1 的「從量測資料建模」連結

**互動流程**
```
1. 資料輸入（二擇一）
   [貼上 CSV 文字] 或 [拖放 .csv 檔案]
   格式：time, output（單輸入步階）或 time, input, output

2. 圖表預覽
   → 即時繪製量測數據（buildSVGChart，散點 + 折線）

3. 選擇識別方法
   [ARX] [ARMAX] [OE] [子空間 N4SID] → 顯示各方法說明 tooltip

4. 設定參數
   na（A 多項式階數）、nb（B 多項式階數）、nk（延遲步數）
   → 「自動選階」按鈕（呼叫 autoARXOrder / autoModelOrder）

5. 執行識別 → StatusBar 顯示進度
   → 結果：殘差圖 + 白化檢驗 badge + 擬合度 %
   → [接受此模型] → 進入系統輸入精靈 Step 3（已填入 ss/tf）
```

**視覺規格**
```
Page：右窗格參數設定，左窗格圖表
資料輸入區：高 120px textarea，虛線 border，拖放高亮（--accent border）
方法選擇：radio 卡片，4 列，每卡片含名稱 + 一行說明
殘差圖：buildSVGChart，高 160px，自相關圖
擬合度：大字 badge，90%+ 綠，75-89% 黃，<75% 紅
```

**依賴模組**
`identifyARX`, `identifyARMAX`, `identifyOE`, `identifySubspace`, `autoARXOrder`, `autoModelOrder`, `residualWhitenessTest`, `crossCorrelationTest`, `buildSVGChart`, `toCSV`

**驗收標準**

### B5-3｜條件數 / 精度告警

**使用者故事**：身為分析研究者，我想知道數值計算是否有精度問題（矩陣近奇異、Cholesky 失敗等），而不是默默得到錯誤結果。

**觸發位置**：StatusBar 的警告狀態；計算結果面板頂部 banner

**告警類型**

| 條件 | 嚴重度 | 訊息 |
|------|--------|------|
| κ(A) > 1e8 | error | 系統矩陣接近奇異，計算結果不可信 |
| κ(Wc) > 1000 | warning | Gramian 條件數高，Cholesky 可能不穩定 |
| 最小 HSV < 1e-10 | warning | Hankel 奇異值接近 0，建議先做最小實現 |
| matRank 不足 | error | 矩陣秩虧損，系統可能不可控/不可觀 |
| Lyapunov 解不正定 | warning | A 可能不穩定或 Q 不半正定 |

**視覺規格**
```
告警 banner：alert() 元件，error/warning variant
  展開：顯示條件數值 + 建議動作連結
  可關閉（但 error 級別每次計算都重新顯示）
StatusBar：常駐顯示最高嚴重度
```

**依賴模組**：`matIsPositiveDefinite`, `matRank`, `matEigenvaluesSymmetric`, `alert()`

**驗收標準**
- [ ] κ > 1e8 的矩陣觸發 error banner
- [ ] 關閉 warning banner 後下次重新計算仍重新顯示

### G1｜單位系統切換

**使用者故事**：身為控制工程師，我習慣用 Hz 而非 rad/s（或相反），想全域切換頻率單位，所有圖表與數值同步轉換，不必心算 ×2π。

**觸發位置**：偏好設定（G4）；圖表右上角單位 chip（快速切換）

**單位群組**
```
頻率：rad/s ↔ Hz（×1/2π）
增益：dB ↔ 倍率（linear）
相位：度 (°) ↔ 弧度 (rad)
時間：s ↔ ms
角頻率刻度：Bode 橫軸隨選用單位重繪刻度標籤
```

**互動規格**
```
全域切換：偏好設定選定預設單位 → 所有頁面套用
局部切換：個別圖表 chip「Hz ▾」可覆寫該圖單位
數值顯示：所有 fmtNum 輸出自動帶單位後綴
轉換正確性：ω[rad/s] = 2π·f[Hz]；|G|[dB] = 20·log₁₀|G|
```

**視覺規格**
```
單位 chip：badge() 樣式，--surface-2 底色，右側 ▾ 10px
切換時：圖表軸標籤淡出淡入 150ms（reduced-motion 則瞬間）
偏好設定：radio group，每個單位群組一列
```

**依賴模組**：`fmtNum`, `fmtDeg`, `fmtDB`, `fmtTime`, `bodeData`, `buildSVGChart`, `formatAxisLabel`

**驗收標準**
- [ ] rad/s ↔ Hz 切換後 Bode 圖橫軸刻度與峰值頻率數值正確（×2π）
- [ ] dB ↔ 倍率切換後增益裕度數值換算正確
- [ ] 局部覆寫不影響其他圖表的全域單位設定

**適用操作**
```

  刪除專案 / 清除所有 localStorage

  重置調參歷史

確認對話框規格：
  標題：明確描述後果「刪除專案『DC Motor』？」
```
